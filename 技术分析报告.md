## 二、🔴 严重问题 (Critical)

### 2.1 大量重复代码 — 产品页面 (P0)

**问题描述：** `pages/product/` 目录下 12 个产品页面结构几乎完全相同，每个文件 404 行，仅数据内容不同。**重复代码总量约 4,848 行，占项目总代码量的 41%。**

**受影响文件：**
- [pages/product/banana.vue](pages/product/banana.vue) (404 lines)
- [pages/product/drama.vue](pages/product/drama.vue) (404 lines)
- [pages/product/human.vue](pages/product/human.vue) (404 lines)
- [pages/product/jimeng.vue](pages/product/jimeng.vue) (404 lines)
- [pages/product/jmdraw.vue](pages/product/jmdraw.vue) (404 lines)
- [pages/product/model.vue](pages/product/model.vue) (404 lines)
- [pages/product/music.vue](pages/product/music.vue) (404 lines)
- [pages/product/ppt.vue](pages/product/ppt.vue) (404 lines)
- [pages/product/resume.vue](pages/product/resume.vue) (404 lines)
- [pages/product/sora.vue](pages/product/sora.vue) (404 lines)
- [pages/product/videoclip.vue](pages/product/videoclip.vue) (404 lines)
- [pages/product/xhs.vue](pages/product/xhs.vue) (404 lines)

**对比证据：** 通过 `diff` 对 banana.vue 和 drama.vue 进行比较，发现仅 SEO 元数据、features 数组内容、featureDetails 数据不同，模板结构、样式、交互逻辑完全一致。

**优化建议：** 使用 Nuxt 动态路由 `pages/product/[slug].vue` (该文件骨架已存在但内容为空)，将产品数据抽取到 `utils/productData.ts` 中，通过 `useAsyncData` 根据 slug 动态加载内容。可将 4,848 行代码缩减为约 250 行，减少 **94.8%** 的重复代码。

### 2.2 硬编码凭据泄露 (P0)

**问题描述：** [pages/demo.vue](pages/demo.vue) 中包含多个演示系统的明文密码，直接暴露在公开源码中：

```js
// 多处出现
account: 'admin', password: '123456'
account: 'demo', password: 'demo123'
```

这些凭据指向 `gmlart.cn`、`cnai.art`、`paper.gmlart.cn` 等站点，其中部分是生产环境。

**受影响行：** demo.vue 第 140-196 行及相关多处

**优化建议：**
- **立即**修改所有暴露的演示账号密码（最高优先级）
- 将演示凭据移至环境变量或从 API 获取
- 在 `.gitignore` 中确认敏感配置文件已排除
- 建议在 CI 中添加 `gitleaks` 或类似工具扫描敏感信息

### 2.3 废弃静态资源未清理 (P0)

**问题描述：** [public/unused/](public/unused/) 目录包含约 **8MB** 的废弃静态资源：

```
public/unused/images/     — 11 files, ~6MB
public/unused/plugin/     — 28 files, ~11MB
public/unused/product/    — 4 files, ~724KB
public/unused/root/       — 5 files, ~728KB
```

这些文件已不再被任何代码引用，但仍保留在仓库中，增加构建体积和 clone 时间。

**优化建议：**
- 删除 `public/unused/` 整个目录
- 确认无代码引用后清理，可减小仓库体积约 8MB

**优化建议：** 持续关注 @nuxt/content v3 正式版发布，升级后回归测试全部内容页面（blog、docs、update）。建议锁定具体版本而非使用 `^` 范围。

## 四、🟡 中优先级问题 (Medium)

### 4.1 TypeScript 类型安全性不足

| 问题 | 位置 | 说明 |
|------|------|------|
| `Record<string, unknown>` 大量使用 | [types/nuxt-global.d.ts](types/nuxt-global.d.ts) | 全局函数类型声明过于宽松 |
| `no-explicit-any: warn` | [eslint.config.js](eslint.config.js) | 允许 any 类型 |
| `NavigationItem` 含 `[key: string]: unknown` | [components/docs/Sidebar.vue](components/docs/Sidebar.vue) | 索引签名破坏类型安全 |
| `queryCollection` 返回 `unknown` | [types/nuxt-global.d.ts](types/nuxt-global.d.ts) | queryCollection 无泛型约束 |
| `definePageMeta` 参数类型为 `Record<string, unknown>` | [types/nuxt-global.d.ts](types/nuxt-global.d.ts) | 无页面元数据类型约束 |

**优化建议：**
- 为 Nuxt Content v3 的 `queryCollection` 定义具体泛型类型
- 收紧 `nuxt-global.d.ts` 中的类型声明
- 逐步消除 `any` 使用
- 为页面元数据、SEO 配置定义类型接口

### 4.2 ESLint 配置与代码规范

**问题描述：**
- ESLint 配置中 `no-console` 仅为 `warn`，`no-debugger` 仅为 `warn`（生产代码应设为 `error`）
- 使用 `@eslint/js: ^10.0.1` 和 `eslint: ^10.1.0`，是新主版本，可能存在插件兼容性风险
- 缺少 `eslint-plugin-import` 等常用规则
- Prettier 配置 `printWidth: 100` 可能导致某些模板行过长

**优化建议：**
- 将 `no-console` 和 `no-debugger` 提升为 `error`
- 添加 `eslint-plugin-import` 规则
- 在 CI 中增加 lint 检查步骤
- 考虑添加 `simple-import-sort` 规则统一 import 顺序

### 4.3 缺乏自动化测试

**问题描述：** 虽然 [package.json](package.json) 配置了 `vitest` 和 `@nuxt/test-utils`，但项目中**没有任何测试文件**。`scripts/` 目录为空（仅有 `test:sitemap` 命令但脚本不存在）。

```json
"devDependencies": {
  "vitest": "^3.0.0",
  "@nuxt/test-utils": "^3.21.0",
  "@vue/test-utils": "^2.4.6",
  "happy-dom": "^20.0.11",
  "jsdom": "^27.3.0"
}
```

测试基础设施已完全配置但未使用。

**优化建议：**
- 为核心工具函数 (utils) 编写单元测试
- 为关键页面组件编写冒烟测试
- 在 CI/CD 中集成测试步骤
- 设定测试覆盖率目标（初期 50%+）

### 4.4 样式实现不一致

**问题描述：**
- 同时存在 Tailwind v4 utility class、内联 style 属性、scoped CSS、以及硬编码的十六进制颜色值
- 硬编码颜色如 `#6E58FF`, `#0F0F12`, `#5A5E6A`, `#0078D6` 散布在组件中
- [tailwind.config.js](tailwind.config.js) 使用 Tailwind v3 的配置语法（`content` 数组），但 `package.json` 安装的是 Tailwind v4 (`^4.1.18`)，存在配置兼容性问题
- `bg-linear-to-b` 等 Tailwind v4 语法与 `bg-gradient-to-br` 等 v3 语法混用

**优化建议：**
- 统一所有颜色为 Tailwind 主题变量或 CSS 变量
- 确认 Tailwind 版本与配置语法的兼容性，升级/统一语法
- 建立设计 token 规范文档
- 考虑使用 `@nuxt/ui` 的 `colors` 配置替代硬编码颜色

### 4.5 数据文件过大

**问题描述：** [utils/pluginData.ts](utils/pluginData.ts) 包含 356 行的硬编码应用数据（id、名称、描述、价格等），数据与代码耦合。

**优化建议：**
- 考虑将产品数据移至 Nuxt Content 集合、JSON 文件或外部 CMS
- 或至少拆分到独立的 `data/` 目录

## 七、架构评估

### 7.1 优势

- ✅ Composition API + `<script setup>` 统一风格，符合 Vue 3 最佳实践
- ✅ Nuxt Content 文件系统路由，内容与代码分离
- ✅ Nuxt UI 组件库提供一致的设计语言和深色模式支持
- ✅ 静态站点生成 (SSG)，首屏加载性能好
- ✅ TypeScript + ESLint + Prettier 代码规范工具链完整
- ✅ SEO 元数据配置较为完善（每个页面有 `useSeoMeta`）
- ✅ JSON-LD 结构化数据在 `app.vue` 中配置
- ✅ 响应式设计，桌面端和移动端适配
- ✅ 使用常量管理 UI 参数（`utils/ui.ts`）

### 7.2 不足

- ❌ 缺少 `composables/` 抽象层，业务逻辑与 UI 组件耦合
- ❌ 产品页面大量重复，违反 DRY 原则
- ❌ 没有测试覆盖
- ❌ 巨型组件和页面文件，违反单一职责原则
- ❌ 硬编码 URL 和凭据遍布代码库
- ❌ 数据与代码混合（pluginData.ts），无数据层抽象
- ❌ 无状态管理方案（未使用 Pinia）
- ❌ 缺少 CI/CD 流程配置
- ❌ 无错误监控和性能监控集成

### 7.3 建议的目标架构

```
├── components/        # UI 组件（每个 ≤ 300 行）
│   ├── landing/       # Landing page 区块组件
│   └── docs/          # 文档相关组件
├── composables/       # 可复用逻辑（新建）
│   ├── useScroll.ts
│   ├── useNavigation.ts
│   └── useTypewriter.ts
├── data/              # 静态数据（新建，从 utils 迁移）
│   └── products.ts
├── pages/
│   └── product/
│       └── [slug].vue # 动态产品页面（替代12个重复页面）
├── utils/             # 纯工具函数
├── types/             # TypeScript 类型定义
└── server/            # 服务端 API（按需）
```

## 九、SEO 与无障碍

### 9.1 SEO 问题

- `nuxt.config.ts` 中的 `<meta description>` 字数约 200 字符，超过推荐的 150-160 字符
- `nuxt.config.ts` 中的 `<meta keywords>` 过于冗长，搜索引擎已不再重视此标签
- [error.vue](error.vue) 404 页面 title 使用 `'404 - 页面未找到'`，缺少站点品牌名
- `favicon.svg` 中使用 `sizes: 'any'`，应指定具体尺寸

### 9.2 无障碍 (A11y)

- [AppFooter.vue](components/AppFooter.vue) 中社交媒体图标缺少有意义的 `<svg>` title/desc
- 部分交互元素可能缺少键盘导航支持（未全面测试）
- `BackToTop` 组件的二维码弹窗在移动端需要手势操作，可能需要额外的无障碍处理

### 9.3 移动端体验

- [components/AppNavigation.vue](components/AppNavigation.vue) 使用自定义的 body scroll lock 逻辑，建议考虑使用 Nuxt UI 内置方案
- [components/BackToTop.vue](components/BackToTop.vue) 在移动端触摸目标可能偏小 (w-10/h-20)
- 部分页面使用 `window.innerWidth` 判断移动端，建议使用 CSS 媒体查询或 `useMediaQuery` composable
